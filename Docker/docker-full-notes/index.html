    üìÖ Day 1 ‚Äî Traditional vs Containerized Deployment

    üèóÔ∏è Traditional Deployment / Old Deployment

    üñ•Ô∏è Virtualized Deployment

    ‚ùå VM Disadvantages:

    • Each VM needs its own OS ‚Üí high CPU, memory, and storage consumption.
    • Higher cost for enterprise OS editions.
    • Can deploy multiple applications in one VM, but no isolation.
    • Middleware conflicts ‚Äî can't run multiple versions (e.g., Java 11 vs Java 17).
    • Environment inconsistencies (e.g., config files, env variables).
    • AMIs can help maintain consistency but still require full OS per VM.

    üî¥ Summary:

    • Each VM ‚Üí full OS ‚Üí high cost and compute usage.
    • Middleware/software compatibility issues.
    • No proper isolation for apps within same VM.

    üì¶ Containerized Deployment

    üß± What is a Container?

    • Includes: App code + dependencies + config files + env variables.
    • Containers share the host OS (no separate OS required).
    • Each container has its own OS libraries ‚Üí interacts with OS kernel.
    • Multiple containers can run on a single host OS.

    üõ†Ô∏è Use Cases:

    • Applications
    • Databases
    • Tools (e.g., Jenkins, Nexus)

    üîê Isolation:

    • Container runtime provides:
    • Separate process space
    • Network space
    • Storage space
    • Mount space
    • High CPU ‚Üí container may hang.
    • Out of memory ‚Üí container is killed by OS (via cgroups).
